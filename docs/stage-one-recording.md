Recording walks is the hardest and most important part of the MVP, because if that layer is unreliable, everything else becomes shaky.

The good news is that your core idea is technically sound: a walk can be recorded as a time-ordered stream of location fixes, with each fix carrying coordinates, timestamp, and quality metadata. Expo’s location stack supports background location updates via `expo-location` and `expo-task-manager`, and Expo explicitly documents background location for both Android and iOS, with the key constraint that recording stops if the user terminates the app. ([Expo Documentation][1])

The first important design decision is this: **do not think of recording as “run a timer every 20 seconds.”** Think of it as **subscribing to the operating system’s location service** and persisting the updates it provides. On mobile platforms, the OS decides when it can deliver location updates efficiently, and those updates are more reliable than trying to fake continuous tracking with your own generic background scheduler. Expo’s location API is built around that pattern, and Android’s own guidance stresses that background location frequency is constrained by the OS for battery reasons. ([Expo Documentation][1])

So the recording engine should be built around a **walk session**. When the user taps Start, the app creates a local walk record, begins a background location task, and appends incoming points to local storage. When the user taps Pause, recording state changes but the session remains open. When the user taps Resume, point capture continues under the same walk. When the user taps Stop, the session is finalized, summary stats are computed, and the walk is queued for sync. This gives you a clean state machine: `idle -> recording -> paused -> recording -> completed`. That is not directly prescribed by the platform docs, but it is the natural product architecture for using Expo background location safely. The platform facts that matter are that background location is supported while the app is backgrounded, and that it stops if the app is terminated. ([Expo Documentation][1])

The second big insight is that **raw track points and derived statistics should be stored separately**. The raw points are your source of truth. From them you later derive total distance, moving time, average speed, breaks, ascent, and cleaned route geometry. This matters because location data is noisy. If you only store precomputed summaries, you lose the ability to improve your algorithms later. A good schema for each point is: timestamp, latitude, longitude, altitude if available, speed if available, horizontal accuracy, and perhaps a source marker. Expo location objects and the underlying platform APIs expose location fixes with these kinds of fields, including altitude and accuracy when available. ([Expo Documentation][1])

For MVP, I would strongly recommend **local-first persistence**. Every point should be written to device storage immediately, not buffered in memory waiting for sync. That means a crash, signal loss, or background suspension is less likely to lose the walk. Expo’s background task system is meant for work that continues while the app is backgrounded, but it does not guarantee your JavaScript runtime is immortal, so local durability matters. A practical design is to store active walks and points in SQLite or another durable local store, then upload only after the walk completes or connectivity returns. Expo does not prescribe SQLite for this, but the background-task model makes local-first storage the robust choice. ([Expo Documentation][2])

On permissions, the hardest part is not technical implementation but **user trust and platform compliance**. On iOS, background location requires proper background capability and the right authorization level; Apple notes that “Always” authorization makes location updates available at any time and enables the system to launch the app quietly for some updates. On Android, background location is tightly restricted for privacy and battery reasons, and foreground-service rules are stricter on newer Android versions, especially Android 14 where the correct foreground service type and permission must be declared. ([Apple Developer][3])

That means your onboarding and permission flow should be deliberate. Do not ask for every permission at app launch. Ask for foreground location first when the user starts a walk. Ask for background location only when the user enables recording that should continue with the screen off or app backgrounded. Explain clearly why you need it: “to keep recording your walk while your phone is locked or in your pocket.” This matches Apple’s guidance around requesting the right level of authorization and Google’s best-practice emphasis on clear justification for background location. ([Apple Developer][4])

The practical limitation you need to build around is simple: **recording is reliable while the app is open or backgrounded, but not after force-quit**. Expo states background location stops if the user terminates the app. On Android, background limits are strict unless the app is using the correct foreground-service pattern. On iOS, background location needs the correct capability and authorization, and behavior depends heavily on permission state. So the app should tell users plainly: recording works with the screen off and phone in pocket, but they should not swipe the app away mid-walk. ([Expo Documentation][1])

On the question of **sampling interval**, a fixed 20-second cadence is a reasonable starting thought, but it should not be your whole model. Walking apps usually do better with a combination of time and distance filtering. If you record too frequently, you waste battery and capture jitter while standing still. If you record too sparsely, you cut corners off the route and lose pause/resume clarity. Android’s documentation is explicit that near-real-time background location significantly affects battery life. So for MVP, I would start with moderate accuracy and a conservative filter, then refine from field testing rather than trying to guess the perfect universal interval. ([Android Developers][5])

A good way to think about it is this: the recorder is collecting **breadcrumbs**, not surveying land. Outdoors, especially for walking, many consecutive points add little value if the user is moving steadily on a straight path. But they become valuable for switchbacks, forks, and pause detection. A robust first version would store all reasonably good location updates the OS delivers during recording, then perform post-processing afterward: drop outliers with poor accuracy, merge stationary jitter, smooth tiny zig-zags, and split elapsed time from moving time. That is the right tradeoff for MVP because post-processing is cheaper to improve than re-engineering track capture after launch. Expo and the platform APIs provide the necessary metadata such as accuracy and altitude to support that later cleanup. ([Expo Documentation][1])

Battery will be one of your biggest real-world constraints. Continuous high-accuracy GPS is expensive. The main levers you have are requested accuracy, update frequency, and what work you do per update. During recording, the background task should do as little as possible: validate the point, persist it, and return. Do not try to compute the full walk summary on every point. Do not sync every point live. Keep the hot path minimal and defer heavy work until pause, stop, or review. Android’s documentation on background location limits exists precisely because power consumption is a major concern, and foreground-service guidance reflects that long-running location work is a special case. ([Android Developers][5])

On GPS reliability with the phone in a pocket or bag: **usually good enough, not perfect**. Apple’s Core Location stack uses a combination of GPS, Wi-Fi, and cellular to determine location, so the phone is not relying solely on a clean satellite lock every second. That means a trouser pocket or jacket pocket is typically fine for outdoor walking traces. But deep valleys, dense tree cover, heavy cloud, and devices buried inside backpacks can degrade quality. So your system should assume occasional bad points and handle them gracefully rather than treating every fix as exact truth. ([Apple Developer][6])

Altitude deserves special caution. It can be present in the location data, but it is often noisier than latitude and longitude. For an MVP, you can store it whenever available and compute simple ascent/descent later, but you should expect inflated elevation gain if you do not smooth it. In other words: keep the field, but do not oversell its precision early. Expo exposes altitude when the underlying platform provides it, so the data is worth capturing even if your first pass at elevation stats is basic. ([Expo Documentation][1])

Break detection is another place where raw data and post-processing matter. A walk session has at least three different time concepts: total elapsed time, moving time, and paused time. The app can explicitly track manual pauses, but it should also infer stoppages from the track itself, for example when consecutive points remain within a very small radius for a sustained period. That gives you a better user result than relying on speed fields alone, because speed values can be missing or noisy. This is an engineering inference rather than a directly documented API rule, but it fits the characteristics of mobile location data described by the platform APIs. ([Expo Documentation][1])

Photos fit cleanly into the recording model if you treat them as **events on the walk timeline** rather than part of the location stream. When the user takes a photo inside the app during recording, store it locally immediately, attach a timestamp, and attach either the current fix or the nearest recorded track point. That makes it easy later to show the photo at the right point on the route without complicating the core tracking loop. Keeping photo capture separate from point capture also reduces the risk that media handling slows down location recording. The permission and media details are separate, but technically this timeline-event model is the cleanest architecture. ([Expo Documentation][2])

One subtle but important product choice is whether to render the map during recording. Since you said you only need maps later for review, that simplifies the recorder substantially. You can build the recording layer without any dependency on basemap rendering. That means no tile downloads, no live map camera, and fewer moving parts while tracking. The active recording screen can be just stats plus session controls. Technically, that is a smart MVP call because it isolates the hardest mission-critical behavior—location capture—from map SDK complexity. The location APIs function independently of map rendering. ([Expo Documentation][1])

If I were appraising the recording subsystem as an MVP, I would say this: it is very feasible, but the success criteria are not “captures a point every X seconds.” They are “does not lose the walk,” “works with the screen off,” “survives poor signal,” “uses acceptable battery,” and “produces a believable route after cleanup.” Expo supports the basic model well, but you must design around platform truth: backgrounded yes, force-quit no; durable local writes yes, naive endless timers no. ([Expo Documentation][1])

My practical MVP recommendation would be: build a recorder around Expo background location and task manager, use a development build rather than Expo Go for real testing, store every accepted point locally with accuracy metadata, keep the background task very light, compute summaries after the walk, and make the UI honest about the “don’t close the app” constraint. Expo’s documentation is clear that proper background location testing, especially on iOS, requires the correct capabilities and a real build path rather than assuming Expo Go will prove everything. ([Expo Documentation][1])
